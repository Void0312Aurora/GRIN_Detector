# 真实数据实验方案（面向当前微透镜缺陷主线）

## 1. 目标

这份方案不是为了“一次性采齐所有真实数据”，而是为了先构建一个**最小可行、可接入当前代码、可验证关键假设**的真实实验闭环。

如果你现在要执行的是“**qDIC / 定量微分相衬显微镜 + WLI / 白光干涉仪**”联合采集，请优先看专门文档：

- `docs/QDIC_WLI_MICROLENS_ACQUISITION_PLAN.md`

当前阶段的目标只有 4 个：

- 验证当前方法在真实采集下是否还能成立。
- 验证“稀疏小缺陷 + 平滑微透镜标准面”假设是否基本成立。
- 验证 `pseudo-poisson + residual` 这条链路需要的符号/梯度假设是否仍然可用。
- 建立后续真实数据采集的命名、元数据、配对和验收规范。

## 2. 当前方法的硬约束

现有主线不是一般性的“任意表面缺陷重建”，而是更窄的任务定义：

- 样本应近似看作：`test = standard + sparse_defect`
- 需要有“标准面”概念，而不是只有单张 topography
- 最好存在 `reference / standard / test` 三者关系
- 缺陷应是局部、稀疏、相对标准面的小残差，而不是大面积纹理或全域粗糙度
- 视场内应主要只有一个微透镜，且口径/aperture 可稳定识别
- 采集参数应稳定，否则物理层和门控阈值没有可比性

如果真实实验一开始就不满足这些条件，那么得到的数据更适合用于“方法失效分析”或“下一代模型设计”，不适合直接拿来验证当前基线。

## 3. 最小可行实验问题

第一阶段建议只回答下面 3 个问题：

1. 同一颗“无明显缺陷”的微透镜重复采集时，重建结果是否接近零缺陷？
2. 同一颗微透镜在“clean / defect”两种状态下，当前方法能否把差异稳定地局部化？
3. QC / gating 指标在真实重复采集上是否稳定，能否区分“正常采集”和“异常采集/异常样本”？

只要这 3 个问题没有回答清楚，就不建议直接扩大采集规模。

## 4. 第一阶段建议采样设计

建议把首轮真实实验拆成 3 组样本。

### A. 重复性组（必须有）

用途：验证系统噪声、对准漂移、照明稳定性。

- 选 3 到 5 颗表观最干净的微透镜
- 每颗在不改变任何光路参数的前提下重复采集 5 次
- 每次都记录完整元数据和原始文件

预期：

- 同一颗 clean lens 的重建结果应接近零缺陷
- QC 指标分布应收敛，可作为 real-val 的初始阈值参考

### B. 配对缺陷组（强烈建议有）

用途：验证当前“标准面 + 稀疏残差”主假设。

- 选 3 到 5 颗初始较干净的微透镜
- 先采集 clean 状态，作为 `standard`
- 再引入可控缺陷后采集 `test`
- 每种状态重复采集 3 次以上

优先考虑可逆或可控的缺陷方式：

- 微粒污染
- 可控轻微附着物
- 人工制造的小划痕或局部刻蚀

不建议第一轮就上“不可逆、不可定位、强破坏”的缺陷，因为那会让配对关系和对照解释变差。

### C. 开放样本组（可选）

用途：验证当前方法的边界，不用于第一轮定量结论。

- 自然缺陷微透镜
- 多种工艺状态样本
- 明显粗糙或非稀疏缺陷样本

这组数据主要用于回答：“当前方法什么时候失效”，而不是回答“当前方法是否成立”。

## 5. 每个可用样本单元的最小组成

对当前代码来说，最有价值的真实数据单元不是单张图，而是一组配对数据。

最低要求：

- `reference`
- `standard`
- `test`

更推荐的实际采集单元：

- `reference` 重复 3 次
- `standard` 重复 3 次
- `test` 重复 3 次

如果当前仪器流程很难稳定拿到 `reference`，也至少要保证：

- 同一颗 lens 的 clean `standard`
- 同一颗 lens 的 defect `test`
- 同光路、同参数、同视场、同尺度

否则后续很难判断误差来自缺陷、采集漂移，还是几何不一致。

## 6. 采集时必须固定的条件

下面这些量必须尽量固定，并写入元数据。

- 仪器型号
- 物镜倍率
- NA
- 波长或有效光源信息
- 像素尺寸或 lateral sampling
- 曝光时间
- 增益
- 照明强度
- 偏振/相移/剪切相关设置
- 样本朝向
- 工作距离
- 环境信息：日期、操作者、温度范围、是否重新校准

第一阶段不要在同一实验集中混用不同倍率、不同光源、不同焦平面策略。

## 7. 视场与样本要求

为了兼容当前主线，建议优先满足：

- 单张视场只包含一个微透镜
- aperture 边界完整，不被截断
- 中心定位稳定
- 视场周围保留少量边缘余量，便于后续 edge-band 分析
- 尽量采集“标准面主形状清晰、边缘不破碎”的样本

如果一个视场里同时出现多个透镜，或 aperture 识别不稳定，这批数据不适合作为当前方法的主验证集。

## 8. 必须记录的元数据

建议每次采集至少记录以下字段：

- `session_id`
- `capture_id`
- `sample_id`
- `sample_role`
- `sample_state`
- `paired_standard_capture_id`
- `paired_reference_capture_id`
- `repeat_index`
- `operator`
- `instrument`
- `objective`
- `na`
- `wavelength_nm`
- `pixel_pitch_um`
- `fov_um`
- `exposure_ms`
- `gain`
- `illumination`
- `focus_strategy`
- `aperture_visible`
- `single_lens_in_fov`
- `raw_file_paths`
- `notes`

这些字段的模板见：

- `external_data/templates/real_capture_manifest.example.json`

## 9. 第一阶段推荐样本规模

建议不要一上来采几百个样本。第一轮只需要足够回答“当前方法能否站住”。

推荐最小规模：

- clean 重复性组：3 颗 lens × 5 repeats = 15 captures
- 配对缺陷组：3 颗 lens × 2 states × 3 repeats = 18 captures
- reference captures：每个 session 至少 3 到 5 份

加起来控制在几十次采集以内即可。

如果这一步通过，再扩充到：

- 更多 lens identity
- 更多 defect type
- 更多 session/day
- 更多设备漂移条件

## 10. 当天采集完成后的快速验收

每个 session 结束前，必须立刻做一次 quick check。

建议最低检查项：

- 原始文件是否都能读
- 元数据是否完整
- 是否能正确分组到 `reference / standard / test`
- clean 重复采集之间是否基本对齐
- 同一颗 lens 的 repeat 之间是否出现大尺度漂移
- 是否有明显过曝、饱和、遮挡、边界截断

如果这些最基本的问题当天不查，后面往往无法补救。

## 11. 与当前代码的接入优先级

建议分两步接入仓库。

第一步：

- 只把真实数据当成 validation 数据
- 不直接用于训练
- 先检查 QC / gating、重建稳定性和失败模式

第二步：

- 再考虑用真实数据做 hybrid dataset
- 或把真实 clean surface 当背景项，叠加 synthetic sparse defect

这比一开始就拿真实数据替换主训练集要稳得多。

## 12. 当前阶段最值得优先验证的 5 条假设

1. clean lens 的重复采集重建是否接近零
2. 稀疏缺陷是否真的是局部残差，而不是大面积表面纹理
3. `standard -> test` 的差分是否足以稳定驱动 `pseudo-poisson`
4. edge artifact 是否会在真实数据上压过真实缺陷
5. 当前 QC 指标是否能在无 GT 情况下筛掉坏样本

## 13. 建议的下一步动作

按顺序做，不要并行发散：

1. 先确认第一轮真实实验使用的仪器与固定参数
2. 选出 3 到 5 颗 clean lens 作为重复性组
3. 设计 1 到 2 种可控缺陷注入方式
4. 按模板开始记录元数据
5. 完成第一轮 pilot 后，再决定是否扩大规模或修改方法假设
